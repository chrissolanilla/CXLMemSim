End-to-end interpretation of your whole setup
1. The test harness creates a hot set

Each round, you do:

for (rep = 0; rep < 200; rep++)
  for (i = 0; i < 8; i++)
    record_access(base + i * 4096);

That means the same 8 pages get a lot of score added every round.

So these 8 addresses become the policy’s strongest hot-page candidates.

2. compute_once() reinforces scores from current placement

Then compute_once() also calls record_access() for:

every page already in the controller
every page in direct expanders

So score is influenced by both:

synthetic accesses from the benchmark
a background “residency refresh” for all currently placed pages

That means your 8 hot pages get a very large boost from the outer loop, while all resident pages get a smaller passive boost.

Then the policy may clear scores if:

current_time - last_cleanup > cleanup_interval

but this depends on controller->last_timestamp actually advancing.

3. get_migration_list() decides whether the local/remote split is wrong

This is the real policy.

It counts:

local_pages = controller->occupation.size()
remote_pages = sum(expander->occupation.size())

Then calculates:

current_fraction_local = local_pages / (local_pages + remote_pages)
Case A: local fraction too small
if (current_fraction_local < target_fraction_local - tolerance)

Then it looks only at remote pages, scores them using access_score, sorts by highest score first, and returns up to batch_size of them.

That means it selects the hottest remote pages for migration.

When perform_migration() processes those addresses, it sees they are in expanders and moves them:

expander → controller

So this is promotion of hot remote pages.

Case B: local fraction too large
if (current_fraction_local > target_fraction_local + tolerance)

Then it looks only at local pages, scores them, sorts by lowest score first, and returns up to batch_size of them.

That means it selects the coldest local pages.

When perform_migration() processes those addresses, it sees they are in the controller and moves them:

controller → expander[0]

So this is demotion of cold local pages.

Case C: close enough to target

No pages are returned, so compute_once() returns 0 and no migration happens.

What this means in plain English

This is a capacity-ratio controller:

It wants the controller to hold about target_fraction_local of all pages.
If the controller holds too little, it fills local memory with the best remote candidates.
If the controller holds too much, it evicts the worst local candidates.
If the ratio is acceptable, it stays still.

So the policy is balancing placement fraction first, and temperature second.

That is why the class name “FractionGuidedMigrationPolicy” fits.

Why perform_migration() now makes sense

Earlier, perform_migration() looked a bit odd because it just moved pages based on where they already lived:

chosen local page → push out
chosen remote page → pull in

Now we can see that this is intentional.

get_migration_list() ensures that:

in “pull in” mode it returns only remote pages
in “push out” mode it returns only local pages

So direction is encoded indirectly by candidate selection.

That is actually a clean separation:

policy decides which addresses
controller performs the move according to current residency
What your 8 hot pages will do

That depends entirely on where they start and whether the current local fraction is below target, above target, or already within tolerance.

Scenario 1: the hot pages start in an expander, and local fraction is too low

Then those 8 pages will likely get very high scores and rise to the top of remote_candidates.

Result:

the hottest of them, up to batch_size, will be pulled into the controller each round
over rounds, more of the hot set should migrate local
once the local fraction reaches the target band, migration stops

This is the most likely “intended” successful behavior.

Scenario 2: the hot pages start in an expander, but local fraction is already near target

Then nothing happens, even if those pages are very hot.

This is important.

The policy does not say “a hot remote page should always be local.”
It says “only migrate when the local/remote fraction is out of range.”

So you can have very hot remote pages that remain remote if the fraction target is already satisfied.

Scenario 3: the hot pages are already local, and local fraction is too high

Then the policy chooses the coldest local pages for eviction.

Since your hot pages have high scores, they should be among the last pages chosen for eviction.

That is good: hot local pages are protected indirectly by score ranking.

Scenario 4: the hot pages are already local, and the local fraction is near target

No migration.

The most important insight

This is not a strict hot/cold migration policy.
It is a fraction-constrained hot/cold policy.

That means the target fraction can dominate the outcome.

For example:

a very hot remote page may stay remote if local fraction is already acceptable
a moderately warm local page may get evicted if the controller is too full and colder pages are insufficient

So the score only decides ordering among candidates in the currently needed direction.

Likely behavior over rounds

Assuming scores accumulate and cleanup does not reset often, each round strengthens the 8-page hot set more and more.

Then one of these patterns emerges:

If local fraction starts below target
Round 1: hottest remote pages start moving local
Round 2+: more hot pages move local until the target band is reached
After that: “no migration triggered”
If local fraction starts above target
cold local pages are pushed out
your hot pages stay local if they are already there
if your hot pages are remote, they will not be pulled in until the local fraction drops below target

So the policy can create a somewhat unintuitive effect:
it may first evict cold local pages for ratio reasons, and only later promote hot remote pages if the ratio crosses the lower bound.

What print_counts(before/after) is probably showing

Now the printed before/after stats have a concrete meaning:

before = current local/remote distribution before adjustment
after = new distribution after up to batch_size pages are moved to restore the target fraction

So you should see page counts changing toward the target band.

If batch_size is small, the ratio should approach the target gradually across rounds.

If batch_size is large, it may jump quickly into the acceptable band.

Design strengths

This policy has some nice properties:

Stable target occupancy

It avoids endlessly chasing every hot page. It only acts when the local share deviates enough from target.

Score-aware promotion

When space should shift local, it chooses the best remote pages.

Score-aware eviction

When local is overfull, it evicts the least valuable local pages.

Hysteresis via tolerance

The tolerance band reduces oscillation. Small ratio fluctuations within the band produce no movement.

Design limitations and quirks
1. It only sees direct expanders

In get_migration_list() and compute_once(), you only scan:

controller->expanders

But perform_migration() can recursively find pages in deeper switch hierarchies.

So pages behind child switches may:

be movable
but not be counted in remote_pages
and not participate in remote_candidates

That is a topology mismatch.

2. Demotion always goes to expanders[0]

So pushing pages out is not load-balanced.

3. size is ignored during actual movement

The migration list stores (addr, size), but only addr is really used in perform_migration().

4. Score refresh may blur true access behavior

Because compute_once() records an access for every resident page, hotness is not purely trace-driven.

5. Cleanup depends on last_timestamp

If last_timestamp is static, scores may never decay, and the policy becomes increasingly history-dominated.

In one sentence

Your full system is a target-local-fraction memory migration policy where the benchmark repeatedly heats 8 pages, compute_once() updates per-page scores, get_migration_list() decides whether the controller has too few or too many local pages, and perform_migration() then either promotes the hottest remote pages or demotes the coldest local pages to bring the local/remote page ratio back toward the configured target.






1. Your access loop and compute_once() both update hotness

This means the hotness scores are not based only on the synthetic trace. They are also biased by residency.

A page can get score increases from:

explicit access generation in the round loop
being present in controller occupation
being present in expander occupation

So hotness is partly “observed traffic” and partly “currently allocated pages.”

That may or may not be what you intended.

If you wanted the policy to reflect only actual memory accesses, the scans inside compute_once() may be too aggressive.
